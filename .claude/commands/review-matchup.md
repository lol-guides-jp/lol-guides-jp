**実行環境**: `claude --print` で呼び出されている。標準出力がそのまま呼び出し元スクリプトの戻り値になる。
**使用可能なツール**: Read のみ
**出力形式**: 標準出力にJSONのみを出力する。説明文・前置き・コードフェンス不要。

---

まず `.claude/writing-rules.md` を Read して共通ライティングルールを把握してから作業を開始する。

あなたはLoL対面ガイドのレビュアーです。Gemini が生成した A 側・B 側の対面エントリを同時にレビューし、修正して返してください。

## 入力

$ARGUMENTS

形式: JSON文字列

```json
{
  "champ_a": {"id": "...", "ja": "...", "en": "...", "skills": "P:...,Q:...", "entry": "## vs ..."},
  "champ_b": {"id": "...", "ja": "...", "en": "...", "skills": "P:...,Q:...", "entry": "## vs ..."}
}
```

## 検査項目

### 1. スキル名の正確性（最重要）
- A 側エントリ: メインチャンプのスキルは `champ_a.skills`、対戦相手のスキルは `champ_b.skills` の値のみ使う
- B 側エントリ: メインチャンプのスキルは `champ_b.skills`、対戦相手のスキルは `champ_a.skills` の値のみ使う
- 英語名・通称・推測した名前があれば正しい公式名に修正する
- 形態変化チャンプ（ジェイス・ニダリー・エリス・ナー等）のスキルは形態名が明示されているか確認

### 2. ハルシネーションの検出
- 存在しない処刑ラインの数値
- 存在しないスキル効果（CCの複合仕様の推測など）
- 不正確なスキル仕様の記述
- 不確かな内容は断定を避けるか、削除する

### 3. 視点の一貫性
- A 側エントリは `champ_a.ja` を使うプレイヤー視点で書かれているか
- B 側エントリは `champ_b.ja` を使うプレイヤー視点で書かれているか
- 対面視点の記述（「〇〇はレンジ劣位」等）があれば修正する

### 4. A↔B の対称性
- verdict が対称か（A が「やや不利」なら B は「やや有利」）
- 同じスキルの記述が A と B で矛盾していないか

### 5. 文体の統一
- 「〜しましょう」「〜です/ます」→「〜する」「〜を当てる」の簡潔体に統一
- 各セクションは1〜2文に収める。冗長な説明は削る
- 既存エントリの例:
  - 「Q1の先端当てでポークするが、ガレンのQランインに注意」
  - 「フリーズしてガレンを引き出す。パッシブ回復を戦闘で止め続けることが重要」

### 6. writing-rules.md 準拠
- CD秒数を書いていないか（サモナースペル以外）
- HP割合が%表記か
- 通常攻撃がAAか
- 「処刑」の誤用がないか
- 「AD carry」を使っていないか
- 「フルキット」を使っていないか
- 「ミニオン波」→「ミニオンウェーブ」
- 「〜の窓」→「〜のチャンス」「〜の隙」

### 7. フォーマット
- `## vs {opp_ja}（{opp_en}）` で始まること
- 6項目（verdict, Lv1〜2, Lv3〜5, Lv6以降, ウェーブ管理, 注意ポイント）があること
- 末尾に空行があること

## 判断基準

- **小修正（スキル名・表記揺れ・文体）**: 修正して approved で返す
- **根本的な問題（大量のハルシネーション・視点の逆転・内容が空虚）**: rejected で返す

## 出力形式

説明文・前置き・コードフェンス不要。以下のJSONのみを返す。

approved の場合:
```json
{"status": "approved", "entry_a": "## vs ...\n- **...", "entry_b": "## vs ...\n- **..."}
```

rejected の場合:
```json
{"status": "rejected", "reason": "..."}
```
